# სერვის მენეჯერების რეპორტი (პროექტი × სართული)

ლისტი 82-დან (Bitrix24) მონაცემების წამოღება და HTML რეპორტის გენერაცია: **პროექტების მიხედვით სართულებზე მომაგრებული სერვის მენეჯერები**.

- მონაცემები იღება **გვერდებად** (50 ჩანაწერი გვერდზე) და **დაყოვნებით** მოთხოვნებს შორის, რათა API არ გაჭედოს.
- რეპორტი იწერება `report.html`-ში (ცხრილი: პროექტები რიგებში, მენეჯერები სვეტებში, უჯრებში სართულების დიაპაზონები).

## გაშვება

1. დააყენეთ [Node.js](https://nodejs.org/) (თუ არ არის).
2. ტერმინალში პროექტის საქაღალდიდან:

```bash
node fetch-report.js
```

3. რეპორტის ნახვა: გახსენით `report.html` ბრაუზერში.

## პირველი გაშვება / ველების შეცვლა

Bitrix24-ში ლისტის ველების ID-ები შეიძლება განსხვავებული იყოს. თუ რეპორტი ცარიელი ან არასწორია:

1. გაუშვით **დებაგ რეჟიმში** (რომ ნახოთ API-ის პასუხის სტრუქტურა):

```bash
node fetch-report.js --debug
```

2. კონსოლში გამოჩნდება პირველი ჩანაწერის ველები. `config.js`-ში `fields` ობიექტში შეცვალეთ ველების სახელები იმ მიხედვით, რასაც API აბრუნებს (მაგ. `PROPERTY_1472` → თქვენი პროექტის ველის ID).

## კონფიგურაცია (`config.js`)

| პარამეტრი | აღწერა |
|-----------|--------|
| `apiBase` | Bitrix24 REST webhook (ბოლოში `/` არ სჭირდება) |
| `listId` | ლისტის ID (82) |
| `pageSize` | ჩანაწერების რაოდენობა ერთ გვერდზე (50) |
| `delayBetweenRequestsMs` | დაყოვნება მოთხოვნებს შორის (ms) – რომ API არ გაჭედოს |
| `fields` | ველების მაპინგი (პროექტი, სართული-დან, სართული-მდე, პასუხისმგებელი) |

## ფაილები

- `fetch-report.js` – მოთხოვნები API-ზე, პივოტი, HTML-ის გენერაცია
- `config.js` – API მისამართი, ლისტის ID, ველების სახელები
- `report.html` – გენერირებული რეპორტი (შეიქმნება გაშვების შემდეგ)
